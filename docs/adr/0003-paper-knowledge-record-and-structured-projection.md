# Represent each paper as a Paper Knowledge Record with a generated Structured Projection

## Context

The project goal has moved beyond single-paper Q&A: the Vault should become a cross-paper academic knowledge base. A plain summary file is not enough because researchers need a readable artifact that captures paper-grounded knowledge, their own thinking, and links to other papers. At the same time, future search, reverse links, graph views, and scripts need stable machine-readable data.

The tempting alternatives are both weak:

- Put everything in Markdown only: good for reading, hard for reliable indexing and graph construction.
- Put knowledge primarily in JSON: easy for scripts, poor for human review and too rigid for evolving research notes.

## Decision

Each paper has a **Paper Knowledge Record**. Its human-readable **Knowledge Surface** is `memory.md`; its machine-readable **Structured Projection** is generated by the plugin as `record.json`.

`memory.md` is the primary user-facing artifact. Its default high-signal sections are:

```text
Abstract
Contribution
Problem
Method
Insight
Results
Takeaways
Reader Thinking
Library Connections
Evidence Pointers
```

`record.json` is generated from `memory.md` and should not be edited by Codex. In V1 it stores parsed Semantic Relationships so scripts, reverse-link views, and graph views can consume stable data without treating Markdown prose as an API.

Semantic Relationships are written in `memory.md` with a parseable line format:

```markdown
- [extends] [Paper title](../OTHERKEY/memory.md): rationale. Evidence: [page 4]
```

Normal sidebar turns use **post-turn review**: Codex may update only the In-Focus Paper's Knowledge Surface during the turn; the plugin then refreshes `record.json`, commits the Vault, and shows update/relationship metadata to the user for review. Pre-commit approval is intentionally not part of this decision because it requires a different patch-generation workflow.

## Consequences

- Researchers keep a readable, editable Markdown surface as the source of durable meaning.
- Scripts and future UI can rely on `record.json` for relationship browsing, reverse links, and graph views.
- Relationship parsing depends on a disciplined Markdown line format; AGENTS.md must keep instructing Codex to use it.
- Schema changes become harder once users accumulate `memory.md` and `record.json`; future migrations need explicit Vault schema versioning.
- Post-turn review is simpler and aligned with the current Codex file-writing model, but it means trust and rollback UI are important follow-up work.
